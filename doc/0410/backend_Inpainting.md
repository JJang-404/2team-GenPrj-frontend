# 인페인팅(Inpainting) 기능 구현 점검 보고서

## 1. 엔드포인트 및 수신 구조 확인 결과
- **현재 상태**: `POST /addhelper/model/changeimage` 호출 시 `ChangeImageRequest` 모델을 사용하지만, **`mask_base64` 필드가 정의되어 있지 않음**.
- **문제점**: 프론트엔드에서 마스크 데이터를 보내더라도 백엔드에서 데이터 바인딩 시 해당 필드가 누락되거나 무시됩니다.

## 2. 백엔드 담당자 확인 사항 (질문 답변)

**Q1. `mask_base64` 파라미터를 받아서 처리하고 있나요?**
> **아니요.** 현재 `ChangeImageRequest` 클래스에 해당 필드가 선언되어 있지 않아 서버에서 수신하지 못하고 있습니다. 인페인팅 기능을 활성화하려면 DTO(BaseModel) 클래스에 해당 필드 추가가 선언되어야 합니다.

**Q2. 내부적으로 사용하는 AI 파이프라인이 img2img인가요, inpainting인가요?**
> 현재 구조는 **img2img** 방식입니다. `mask_base64`가 전달되지 않고 `strength` 값만 사용되고 있기 때문에 이미지 전체를 변형하는 방식으로 동작합니다. 인페인팅으로 전환하려면 업스트림 엔진(`engine_url`) 호출 시 `mask_image` 인자를 함께 전달하는 로직이 추가되어야 합니다.

**Q3. 마스크 컨벤션이 어떻게 되나요?**
> 현재 백엔드에 마스크 처리 로직이 없으므로 컨벤션이 확정되지 않았습니다. 일반적으로 Stable Diffusion 계열은 **흰색(255) 영역을 재생성 영역**으로 처리하므로, 프론트엔드의 규칙(흰색=배경/생성)과 일치할 것으로 보이나, 실제 파이프라인 적용 시 `invert_mask` 여부를 검증해야 합니다.

## 3. 향후 조치 권장 사항
1. **모델 클래스 수정**: `app/models/openai.py`의 `ChangeImageRequest`에 `mask_base64` 필드 추가.
2. **업스트림 요청 수정**: 백엔드가 호출하는 실제 엔진(SD 파이프라인) 단에서 `mask_image` 파라미터를 지원하는지 확인하고, 수신된 `mask_base64`를 디코딩하여 전달하도록 수정.
3. **API 문서 업데이트**: `docs/API.md`에 `mask_base64` 페이로드 규격 추가.
