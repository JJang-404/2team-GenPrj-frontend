# sample_ver3 설명용 문서
목표는 두 가지다.

1. 왜 `sample_ver3`가 지금 구조로 설계되었는지 이해시키기
2. 첨부된 Lean Canvas 메모와 아이디어를 실제 서비스 구조로 연결해서 설명하기

---

## 1. 우리가 만들려는 것이 정확히 무엇인가

이 프로젝트는 단순히 "포스터 한 장 생성기"가 아니다.  
방향은 아래에 가깝다.

- 가게 정보와 상품 사진을 입력하면
- 광고 문구와 디자인 배치를 자동으로 만들고
- 필요하면 이미지 생성 모델도 붙여서 비주얼을 보강하고
- 이후 사용자가 자연어로 수정 요청까지 할 수 있는
- 광고 제작 보조 시스템

즉, 핵심은 **이미지 생성만 잘하는 시스템**이 아니라,
**입력 정보 -> 광고 기획 -> 템플릿 선택 -> 렌더링 -> 수정**까지 이어지는 구조다.

첨부 메모에서도 이 부분이 반복된다.

- 가게 정보 입력이 중요하다
- 사업 컨설팅 수준으로 세부 정보를 받을수록 결과가 좋아진다
- 문구와 배치가 중요하다
- 사용자는 모바일보다 웹에서 편하게 편집할 가능성이 크다
- 최종적으로는 "광고 + 컨설팅" 성격이 있어야 한다

그래서 `sample_ver3`는 단순 생성 모델 중심이 아니라 **안전한 광고 생성 파이프라인** 중심으로 설명하는 것이 좋다.

---

## 2. 팀원들에게 설명할 때 가장 먼저 말해야 할 포인트

### 한 줄 설명

`sample_ver3`는 "LLM을 붙였지만, LLM이 실패해도 광고 생성이 무너지지 않도록 만든 구조"다.

### 더 쉬운 설명

- `ver1`의 장점: 안정적이다
- `ver2`의 장점: 유연하고 확장성이 크다
- `ver3`의 목표: 둘을 합친다

즉:

- LLM이 잘 되면 더 똑똑하게 동작
- LLM이 실패하면 규칙 기반으로 안전하게 동작
- 검증되지 않은 템플릿은 메인 경로에 올리지 않음

이렇게 설명하면 팀원들이 `ver3`의 존재 이유를 바로 이해한다.

## 3. 첨부 이미지 내용을 반영했을 때의 제품 정의

첨부 이미지의 Lean Canvas를 보면 실제 서비스 정의는 아래처럼 정리할 수 있다.

### Problem

- 소상공인은 디자인 품질이 낮거나 직접 만들 시간이 부족하다
- 이미지 생성 AI만 써서는 광고 문구, 배치, CTA까지 잘 안 나온다
- 가게 정보와 메뉴 정보가 빠지면 결과 품질이 떨어진다

### Solution

- 이미지 기반 광고 생성
- 자동 문구 생성
- 템플릿 기반 배치
- 자연어 수정 요청
- 가게 정보 기반 추천/컨설팅

### Unique Value Proposition

- 사진 한 장과 가게 정보만으로 광고를 빠르게 만든다
- 단순 이미지 생성이 아니라 "광고용 결과물"을 만든다
- 사용자는 디자인 툴을 몰라도 된다

### Unfair Advantage

- 가게 정보 입력을 많이 받을수록 더 좋은 결과를 만들 수 있다
- 이미지, 문구, 배치를 한 구조 안에서 다룬다
- 추후 컨설팅 성격까지 붙일 수 있다

### Customer Segments / Early Adopters

- 카페, 디저트 가게, 브런치, 음식점
- 디자인 툴 사용이 익숙하지 않은 소상공인
- SNS 홍보/신메뉴 홍보가 필요한 자영업자

### Channels

- 우선순위는 웹
- 이후 챗봇/업로드 중심 인터페이스 확장 가능

### Revenue Streams

- 광고 생성 사용량 기반
- 광고+컨설팅 패키지
- 템플릿 팩 / 업종 특화 팩

### Cost Structure

- GPU 비용
- API 비용
- 서버 운영 비용

즉, 팀원들에게는 이 프로젝트를 "광고 이미지 생성기"가 아니라 **소상공인 광고 제작 도우미 플랫폼의 백엔드 구조**로 설명하는 게 맞다.

## 4. 왜 입력 구조가 중요하고, 어떤 입력을 받아야 하는가

첨부 메모에서 가장 중요한 포인트 중 하나는 이것이다.

> 입력 요소가 많을수록 결과 품질이 올라간다.

이건 `sample_ver3` 구조 설명에서도 꼭 강조해야 한다.

### 최소 입력

- 업로드 이미지
- 업종
- 홍보 목표
- 원하는 분위기

### 권장 입력

- 가게명
- 대표 메뉴
- 가격대
- 시그니처 메뉴 여부
- 운영 시간
- 위치
- 주차 가능 여부
- 시즌 정보
- 할인 여부
- 행사 정보
- 강조하고 싶은 문구

### 왜 필요한가

이미지 생성 모델만으로는 아래가 안 나온다.

- 문구 우선순위
- 어떤 CTA를 넣어야 하는지
- 어떤 정보를 크게 보여야 하는지
- 어떤 업종/타겟에 맞는 템플릿인지

즉, 입력은 단순 참고값이 아니라 **광고 기획의 재료**다.

그래서 구조적으로는 입력을 다음 두 레이어로 나누는 것이 좋다.

### Layer A: 생성 입력

- 이미지
- 톤
- 목표
- 비즈니스 타입

### Layer B: 컨설팅 입력

- 위치
- 가격대
- 운영 시간
- 대표 상품
- 시즌성
- 주차 가능 여부
- 방문/배달 여부
- 타겟 고객

이렇게 나누면 팀원들이 "왜 입력 폼이 길어질 수밖에 없는지"를 이해하기 쉽다.

## 5. 첨부 메모를 반영했을 때 가장 좋은 전체 구조

서비스 관점에서 설명하기 좋은 구조는 아래다.

```text
사용자 입력
-> 입력 정규화
-> 광고 기획 레이어
-> 템플릿 선택 레이어
-> 비주얼 생성/보강 레이어
-> 최종 렌더링 레이어
-> 수정 요청 레이어
-> 프론트 편집 레이어
```

이걸 `sample_ver3` 코드 기준으로 설명하면 더 명확하다.

### 1. 입력 정규화

역할:

- 사용자 입력을 정리한다
- 업종, 목표, 분위기, 필수 정보 누락 여부를 체크한다
- 입력이 부족하면 어떤 정보가 더 필요한지 판단한다

이 레이어는 앞으로 반드시 필요하다.  
현재 `sample_ver3`는 선택/수정/검증 중심 구조지만, 실서비스로 가려면 이 단계가 앞에 와야 한다.

### 2. 광고 기획 레이어

역할:

- 어떤 정보를 headline으로 올릴지
- 가격을 강조할지
- CTA를 넣을지
- 메뉴판형인지 프로모션형인지
- 고급형/일반형/빈티지형 중 어디가 맞는지

즉, 이 레이어는 "광고 문구와 우선순위 설계"를 담당한다.

첨부 메모에서 말한 "컨설팅 모드"는 사실 이 레이어에 해당한다.

### 3. 템플릿 선택 레이어

현재 `sample_ver3`에 이미 들어간 구조다.

역할:

- 검증된 템플릿만 후보로 사용
- LLM이 있으면 더 유연하게 선택
- 없으면 룰 기반 fallback

관련 파일:

- `ai/template_catalog.py`
- `ai/template_loader.py`
- `ai/template_selector.py`

### 4. 비주얼 생성/보강 레이어

첨부 메모에서 말한 "두 가지 갈래"를 여기서 설명하면 된다.

#### 갈래 A: 기존 입력 이미지를 그대로 활용

- 사용자가 올린 사진을 중심으로 광고를 만든다
- 가장 안정적이다
- 현재 기본 경로는 이쪽이어야 한다

#### 갈래 B: 생성 모델로 비주얼을 추가 생성/보강

- Diffusers / SDXL / ControlNet 계열
- 이미지 품질을 끌어올릴 수 있음
- 하지만 환경 의존성과 실패 가능성이 크다

여기에 첨부 메모의 "디퓨전 생성 + GPT API 추가 생성 가능" 아이디어를 연결하면 된다.

즉, 설명은 이렇게 하면 좋다.

- 기본 경로는 안전한 이미지 기반 광고 생성
- 고급 경로로 이미지 생성 모델을 붙인다
- 이미지 생성과 문구/배치는 분리해서 본다

### 5. 최종 렌더링 레이어

역할:

- 텍스트
- 이미지 슬롯
- 가격 배지
- 리본/뱃지
- 배경

를 실제 결과물로 합성한다.

이 단계는 deterministic 해야 한다.  
즉, 최종 렌더러는 가급적 LLM에 직접 의존하지 않고, 검증된 JSON만 받아서 그리는 구조가 좋다.

### 6. 수정 요청 레이어

현재 `sample_ver3`에 이미 들어간 구조다.

역할:

- "가격 더 크게"
- "좀 더 고급스럽게"
- "텍스트 줄여줘"
- "배경 밝게"

같은 요청을 받아 반영한다.

중요한 점:

- 규칙 기반 수정이 항상 바닥에 있어야 한다
- LLM은 그 위에 덧붙는 보조 해석기여야 한다

관련 파일:

- `ai/revision_parser.py`
- `ai/validation.py`

### 7. 프론트 편집 레이어

첨부 메모에 있던 "사진 에디팅 서비스를 하려면 프론트엔드 기능이 하이레벨로 나와야 한다"는 말은 매우 중요하다.

실제로는 사용자가 아래를 직접 만지고 싶어할 가능성이 높다.

- 텍스트 위치
- 폰트 크기
- 색상
- 배지 크기
- 전화번호/가격 강조
- 문구만 교체

그래서 최종 구조는 아래처럼 가는 것이 좋다.

```text
백엔드가 초안을 만든다
-> 프론트엔드 편집기가 마지막 조정을 맡는다
```

이 구조가 팀원들에게 가장 현실적으로 전달된다.

## 6. 팀 설명용으로 가장 좋은 모듈 구분

팀원들에게 설명할 때는 기술 파일명보다 역할 중심으로 나누는 게 좋다.

### A. Input Layer

해야 할 일:

- 업로드 이미지 받기
- 가게 정보 받기
- 필수 정보 체크
- 부족한 정보 추가 질문

추천 책임:

- form schema
- request validator
- input normalizer

### B. Planning Layer

해야 할 일:

- 광고 목적 해석
- 어떤 문구를 강조할지 결정
- 메뉴판형/프로모션형/브랜딩형 분기

추천 책임:

- prompt planner
- copy planner
- CTA planner

### C. Template Layer

해야 할 일:

- 사용 가능한 템플릿 목록 관리
- 템플릿 메타데이터 정리
- 렌더 가능 여부 검증
- 템플릿 선택

현재 `sample_ver3`의 강점은 이 부분이다.

### D. Visual Layer

해야 할 일:

- 원본 이미지 사용
- 필요 시 생성 모델 호출
- 이미지 전처리/보정

### E. Render Layer

해야 할 일:

- 최종 광고 이미지 합성
- 고정된 규칙으로 안정적 출력

### F. Revision Layer

해야 할 일:

- 자연어 수정 요청 해석
- override 생성
- 안전한 검증 후 반영

### G. Editor Layer

해야 할 일:

- 프론트에서 세부 위치/크기/색상 수정
- 최종 저장

## 8. 팀원들에게 설명할 때 강조해야 할 개발 원칙

### 원칙 1. LLM은 중심이 아니라 보조다

LLM이 잘하면 품질이 올라간다.  
하지만 실패해도 기본 동작은 유지돼야 한다.

### 원칙 2. 최종 렌더는 deterministic 해야 한다

최종 결과물을 그리는 레이어는 예측 가능해야 한다.  
그래야 디버깅과 수정이 가능하다.

### 원칙 3. 검증되지 않은 템플릿은 메인 경로에 올리지 않는다

자동 생성 템플릿은 많아도, 검수 전까지는 후보군에서 제외하는 것이 맞다.

### 원칙 4. 입력을 많이 받는 것은 불편이 아니라 품질 전략이다

이 서비스는 가게 정보가 많을수록 결과가 좋아진다.  
즉, 긴 입력 폼은 설계 결함이 아니라 품질 전략일 수 있다.

### 원칙 5. 최종 UX는 웹 편집기로 닫힌다

생성만으로 끝나지 않는다.  
사용자는 마지막에 직접 조정하고 싶어한다.

## 10. 발표나 회의에서 이렇게 설명하면 좋다

짧게 설명하면 아래 흐름이 가장 좋다.

### 설명 순서

1. 우리는 단순 포스터 생성기가 아니라 소상공인 광고 제작 보조 시스템을 만들고 있다.
2. 그래서 입력은 사진만이 아니라 가게 정보까지 포함해야 한다.
3. `ver1`은 안정적이고 `ver2`는 유연했지만, `ver3`는 LLM이 실패해도 안전하게 가도록 만들었다.
4. 구조는 입력 정규화 -> 광고 기획 -> 템플릿 선택 -> 비주얼 생성/보강 -> 렌더링 -> 수정 -> 프론트 편집 순서로 가는 것이 맞다.
5. 자동 생성 템플릿과 LLM은 보조 기능이고, 최종 렌더와 핵심 경로는 deterministic 해야 한다.

이렇게 설명하면 기술적인 방향, 제품적인 방향, 실제 구현 이유가 한 번에 연결된다.

## 11. 최종 결론

첨부된 이미지 내용을 반영해서 팀원들에게 설명하려면, `sample_ver3`를 아래처럼 이해시키는 것이 가장 좋다.

- 광고 제작 보조 플랫폼의 백엔드 구조
- 입력이 많아질수록 좋아지는 품질 중심 구조
- LLM을 쓰지만 LLM 실패를 기본적으로 대비하는 구조
- 자동 생성 템플릿은 검수 후 사용하는 구조
- 최종적으로는 웹 편집기와 연결되는 구조

즉, 이 프로젝트의 핵심은 "AI가 예쁘게 한 장 만들어준다"가 아니라,
**"광고 기획과 결과물 생성을 안전하게 연결하는 시스템"** 이라고 설명하는 것이 가장 정확하다.
